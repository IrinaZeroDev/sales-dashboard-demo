# Интерактивный дашборд продаж

Веб-прототип BI-дашборда для менеджеров по продажам. Полный пайплайн на Python:
генерация данных → очистка и предобработка → интерактивная визуализация с динамическими
фильтрами. Прототип построен на Plotly Dash и служит для быстрого согласования состава
метрик и логики фильтров перед переносом в Power BI.

## Приоритетные метрики (по ТЗ)

- Объём продаж (выручка)
- Количество транзакций
- Средний чек
- Продажи по регионам и по категориям товаров

Все они вынесены в KPI-карточки и графики на дашборде.

## Возможности

- **KPI-карточки** с дельтой к предыдущему сопоставимому периоду: объём продаж,
  маржа, транзакции, средний чек.
- **Динамические фильтры**: период (календарь + пресеты «посл. месяц / квартал / год»),
  регион, категория продукта, сегмент клиента (RFM), тип клиента.
- **Гранулярность тренда** на лету: день / неделя / месяц.
- **Переключатель учёта аномалий** (выбросов по выручке).
- Графики: динамика выручки, выручка по регионам, тепловая карта «регион × категория»,
  топ-12 продуктов (ABC), выручка по сегментам клиентов (RFM).

## Источник данных

Excel-файл `sales_raw.xlsx` содержит три листа (как у заказчика — справочники в
отдельных таблицах, не в одной):

| Лист | Содержание | Ключ |
|---|---|---|
| Продажи | таблица фактов: order_id, order_date, client_id, product_id, quantity, unit_price, discount, revenue, cost, manager | client_id, product_id |
| Клиенты | справочник: client_id, client_name, client_type, region | client_id |
| Продукты | справочник: product_id, product_name, category, base_price, base_cost | product_id |

Регион продаж берётся из справочника клиентов — поэтому merge со справочником
обязателен для аналитики по регионам.

## Запуск

```bash
pip install -r requirements.txt
python generate_data.py     # data/sales_raw.xlsx (листы Продажи/Клиенты/Продукты)
python clean_data.py       # data/sales_clean.csv + отчёт
python app.py              # http://127.0.0.1:8050
```

## Методы аналитики данных

**1. Профилирование и контроль качества.** Анализ пропусков, дубликатов, типов,
диапазонов. Генерируется текстовый отчёт (`cleaning_report.txt`) с метриками качества
на каждом этапе.

**2. Парсинг дат.** Исходные даты приходят в форматах `%Y-%m-%d`, `%d.%m.%Y`,
`%d/%m/%Y`, `%d %b %Y`. Формат определяется по разделителю, чтобы ISO-даты не ломались
глобальным `dayfirst` (`2025-12-04` должно остаться 4 декабря, а не 12 апреля).

**3. Нормализация справочников.** Регионы и категории в справочниках приводятся к
каноническому виду: strip, нижний регистр, замена опечаток («0» вместо «о»), маппинг
в справочник. Это сводит десятки вариантов написания к каноническим.

**4. Merge факта со справочниками.** LEFT JOIN таблицы фактов со справочниками
клиентов и продуктов по `client_id` и `product_id`. Сделки с «висящими» ключами
(которых нет в справочнике) не теряются — `client_name`/`product_name`/`region`/
`category` заполняются значением «Не определено», чтобы подсветить проблемы качества
данных и не потерять выручку.

**5. Обработка пропусков.** Категориальные (`region`, `client_type`) → «Не определено».
Числовые (`revenue`) → восстановление по формуле `quantity × unit_price × (1 − discount)`.

**6. Поиск аномалий.** Метод IQR (Q1, Q3, upper = Q3 + 1.5·IQR) и Z-оценка
(|z| > 3). Выбросы не удаляются, а помечаются флагом `is_outlier` — менеджеры видят
«подозрительные» сделки и могут отключить их фильтром.

**7. Производные признаки.** Маржа и маржинальность (%), средний чек, временные
измерения (год / месяц / квартал / неделя / день недели).

**8. Сегментация клиентов (RFM).** Recency, Frequency, Monetary → квартили →
сегменты: VIP, Постоянные, Спящие, Уходящие.

**9. ABC-анализ продуктов.** Ранжирование по вкладу в выручку с накопительной долей:
A (до 80%), B (80–95%), C (95–100%).

**10. Временные ряды.** Скользящие средние, ресемплинг по дню/неделе/месяцу,
сравнение с предыдущим сопоставимым периодом (дельта KPI).

## Подходы к визуализации

- **Принцип 5 секунд**: ключевые KPI — крупно сверху, с цветной дельтой
  (зелёный ▲ / красный ▼) для мгновенного понимания динамики.
- **Правильный тип графика под задачу**:
  - линейный график с заливкой — динамика и тренд;
  - горизонтальный bar — сравнение регионов и продуктов (удобно читать длинные подписи);
  - тепловая карта — двумерная концентрация выручки «регион × категория»;
  - цветовое кодирование ABC (зелёный/оранжевый/серый) и RFM.
- **Динамические фильтры (slicers)**: все графики и KPI пересчитываются по выбранным
  периоду, региону, категории, сегменту и типу клиента — через Dash callbacks.
- **Минимум шума**: убраны лишние оси/легенды, единая цветовая палитра, акцентный цвет
  только для важного (тренд, аномалии).
- **Адаптивность подписей**: `automargin`, сокращение длинных названий с полным
  именем в hover, пресеты периодов.
- **Доступность**: контрастный текст, читаемые шрифты, hover-tooltips с деталями.

## Перенос в Power BI

1. Импорт `data/sales_clean.csv` как таблицы фактов, `data/dim_date.csv` — как
   измерение дат. Связь по дате → модель «звезда».
2. Повторить меры на DAX, например:
   ```dax
   Выручка = SUM(sales_clean[revenue])
   Маржа = SUM(sales_clean[margin])
   Средний чек = DIVIDE([Выручка], DISTINCTCOUNT(sales_clean[order_id]))
   ```
3. Slicers: `order_date` (диапазон), `region`, `category`, `client_segment`,
   `client_type`. Визуалы переносятся 1:1: карта KPI, линейный график, горизонтальный
   bar, матрица-тепловая карта (Matrix + условное форматирование), bar по продуктам.
4. ABC/RFM можно считать в Power Query или взять готовые флаги из `sales_clean.csv`.

## Данные

Датасет синтетический (сгенерирован `generate_data.py`): ~6 800 сделок за 12 месяцев
(12.07.2025 – 11.07.2026), 250 клиентов, 4 категории и 17 продуктов, 11 регионов РФ.
Справочники клиентов и продуктов лежат на отдельных листах Excel. В сырые данные
намеренно внесены дефекты для демонстрации очистки: дубликаты, пропуски, выбросы,
некорректные количества, даты в разных форматах, опечатки в регионах/категориях,
«висящие» ключи.
При работе с реальными данными достаточно положить ваш Excel-файл как
`data/sales_raw.xlsx` (с листами `Продажи`, `Клиенты`, `Продукты` и теми же колонками)
и запустить `clean_data.py` → `app.py` без изменения кода. Если названия листов или
колонок отличаются — правка делается в `load_workbook()` и `merge_fact()`.
